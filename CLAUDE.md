# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Godot 4.7 (.NET/C#) project building a native asset-import pipeline for **Demon's Souls (PS3)**, reading FromSoftware's proprietary formats (FLVER0 meshes, TPF textures) directly into Godot resources. Project display name is "Soulbrandt"; the C# assembly/csproj is `Soulbrandt` too (`project.godot`'s `[dotnet] project/assembly_name` matches) — renamed from an old `Boletaria` codename, kept only as flavor text in a few in-code comments referring to the *in-game* kingdom of Boletaria, unrelated to the project name.

No game assets are bundled in this repo. `mounted` (gitignored) holds the user's own game copy, populated by an in-editor mount action — see "Asset mounting" below.

The repo (`github.com/Solaire9886/Soulbrandt`) is public and licensed **GPLv3** — not an arbitrary choice: `SoulsFormatsNEXT` (see below) is GPLv3 with no linking exception, and this project links against it directly, so the combined distributed work must be GPLv3 too. `README.md` carries an RPCS3-style disclaimer (no proprietary assets, no piracy support, no DRM-bypass methods); `CONTRIBUTING.md` covers the contributor workflow.

**Read `context.md` and `PLAN.md` too, before making changes here.** `context.md` has the
session history behind this architecture: the real bugs already found and fixed (with
how they were confirmed, not just guessed), C#/Godot interop gotchas, this machine's
environment quirks, and a list of dead ends already tried and rejected. `PLAN.md` has
the roadmap — this asset importer is Phase 1 of a larger goal (a full Demon's Souls
recreation that bundles no proprietary assets, relying entirely on the user's own copy),
which is why some things here (the mounting system especially) are built the way they
are. This file (`CLAUDE.md`) says what's true now; the other two say why and where it's
headed.

## Build & verify

Godot's own C# tooling (`GodotTools`, used by the editor's build/import machinery) hardcodes a `net8.0` requirement with no cross-major rollForward. On Arch, `pacman -S dotnet-sdk-8.0` (from `extra`) satisfies this alongside whatever newer SDK you already have installed, with no manual `DOTNET_ROOT` override needed — plain `dotnet`/`godot-mono` on `PATH` resolves it. On other distros, install whatever your package manager calls the .NET 8 SDK the same way. (An earlier revision of this doc had you build a manual merged root at `~/.dotnet-godot`, under the mistaken assumption that Arch didn't package a net8 SDK at all; confirmed obsolete and dropped — see `context.md`'s "Environment specifics" section for the full history.)

**Build the C# assembly** (do this after any change under `addons/archstone/`):
```
dotnet build Soulbrandt.csproj
```
Do **not** use `godot-mono --headless --build-solutions` to build — it reliably hangs in this environment. Plain `dotnet build` against `Soulbrandt.csproj` is the proven path.

**Force a reimport** (Godot's incremental importer has no idea when the *importer code itself* changed — only source-file mtime — so after editing `FlverSceneImporter.cs` you must invalidate the cache manually, not just rebuild):
```
rm -f .godot/imported/*.flver-*
find mounted -iname "*.flver" -o -iname "*.tpf" | xargs touch   # only if testing map/chr assets
godot-mono --headless --editor --path . --import
```
A full reimport of all ~3400 mounted FLVER files (now spanning `chr`/`map`/`obj`/`parts`, not just `chr`/`map` as before the "Asset mounting" system existed) is expected to end with a small number of errors, all the same known bug: a FLVER0 vertex/UV layout the importer doesn't handle yet (`ArgumentOutOfRangeException` on `v.UVs[0]`/`v.Normals[0]`, `FlverSceneImporter.cs:121`/`130`). Confirmed affected files as of this writing: `m9999b0.flver`/`m9900.flver` across several map areas, plus `o9996.flver` under `obj/` — the first `obj` file ever importable, since `obj` had no loose files at all before the mount system landed. Expect this affected-file set to grow slightly as more of the game becomes newly extractable; the important check is that it's always *this same* exception at *this same* line, not a new failure mode. A genuinely different error (different exception type, different stack, or a huge jump in count) is a regression worth investigating.

**There is no unit test suite.** Verification is done with throwaway `SceneTree`-extending GDScript scripts run headlessly, e.g.:
```
godot-mono --headless --path . -s check.gd
```
where `check.gd` does `load("res://mounted/chr/c9999/c9999.flver").instantiate()` and inspects the resulting node tree (surface count, `surface_get_material(i).albedo_texture`, etc.). Write these to a scratch location and delete them when done — none should be committed.

A bulk reimport occasionally dies mid-batch with `Fatal error. Invalid Program: attempted to call a UnmanagedCallersOnly method from managed code.` — re-running the import (possibly a couple of times) has always recovered without a permanent regression so far; treat as a known flake in Godot's threaded import host, not a code bug, unless it starts happening every run.

## Architecture

**The importer** (`addons/archstone/FlverSceneImporter.cs`) is a Godot `EditorSceneFormatImporter`, registered by `addons/archstone/archstone.gd`'s `_enter_tree()` via `load(...).new()` + `add_scene_format_importer_plugin()`. This goes straight from `.flver`/`.tpf` bytes to native `ImporterMesh`/`StandardMaterial3D` (or `ShaderMaterial` for terrain-blend materials, see below)/`ImageTexture` at import time — there is no intermediate OBJ/MTL/PNG file relay. (`desflver_test/`, an old standalone reference console app used early on for manual FLVER-parse comparison, has been removed — everything it covered now lives in the importer and this doc; its OBJ-export quirks, V-flip and triangle winding swap, never applied to the native importer anyway — see the FLVER↔Godot conventions below.)

Per-mesh geometry is built as plain C# arrays (position/normal/color/UV/index) and handed to the engine in one `SurfaceTool.CreateFromArrays()` call, rather than a per-vertex `SurfaceTool.AddVertex()`/`SetNormal()`/`SetUV()`/... loop — same per-vertex math, far fewer engine-boundary calls. `GenerateTangents()` still runs on the resulting `SurfaceTool` afterward; that call has no array-based equivalent.

**Texture resolution tries an ordered chain of candidate directories per texture reference** (`CandidateDirs`/`ResolveTexture` in `FlverSceneImporter.cs`), because Demon's Souls doesn't use one convention across categories, containers, or even individual materials on the same model. All four rules share one cache/merge primitive, `GetMergedTextures`/`LoadDirTextures` backed by `_dirTextureCache` (a `ConcurrentDictionary<string, Dictionary<string, TPF.Texture>>` keyed by resolved directory — required, not just cautious, because Godot's bulk reimport dispatches `_ImportScene` across multiple worker threads against the same importer instance; a plain `Dictionary` here silently loses entries under contention), which merges *every* `.tpf` found in a directory rather than opening one guessed-basename file — a folder can hold more than one real tpf side by side (e.g. `WP_A_1503.tpf` and a separate `WP_A_1503_L.tpf` variant), and the entry a reference needs isn't always in the one whose name matches the folder. Corrupt/zero-byte `.tpf` files (3 confirmed real cases in the shipped game data: `o3104`, `o0050`, `o7999` — not an extraction bug) are skipped with a `GD.PushWarning` rather than aborting the whole model's import. Texture-name lookups are case-insensitive throughout. In trust order:
1. **The model's own real container** (`OwnModelDir`) — same-basename `.tpf` flattened next to the `.flver` for chr (`mounted/chr/c2000/c2000.tpf`, no `tex` subfolder in the real container at all), or a sibling `tex/` folder for obj/parts, whose container instead splits the flver into its own `sib/` folder (`mounted/obj/o0211/sib/o0211.flver` + `mounted/obj/o0211/tex/o0211.tpf`) — confirmed via directory scan across every mounted obj/parts model, not an edge case. Always correct when the reference actually lives here, since it's exactly what got extracted for this specific model.
2. **The texture's own reference path** (`RefPathDir`) — parses the `Model/.../tex/` segment out of the reference string itself (a dev-tool path, not something the game re-validates at runtime) relative to the mounted root, nested-`tex/`-then-flat fallback. Correct for ordinary map pieces (whose per-area bucket, e.g. `mounted/map/m03/`, holds ~19 numbered `.tpf` files shared by every piece in that area) and for genuine cross-category reuse — e.g. `m4020b0.flver` (a Boletarian Palace corpse pile) uses `chr/c2000`'s own NPC texture set directly, resolved by this same rule with no separate code path needed.
3. **A sibling texture slot on the same material** (`SiblingMapAreaDir`) — if another reference on the same material resolves under `mounted/map`, tries that map-area folder for this one too. Recovers a stale/copy-pasted ref that still names the model's own folder while the real texture lives in a shared map-area bucket — confirmed on all six Nexus archstones (`o1000`-`o1050`): every one shares a single `m01` body texture, correctly labeled only on the `g_Lightmap` slot, with `g_Diffuse`/`g_Specular`/`g_Bumpmap` all copy-pasted from the `o1000` template and never updated.
4. **A map-area prefix baked into the filename** (`MapPrefixDir`, regex `^(m\d\d)_`) — tried even when every reference on the material lies about its own category. Confirmed on `o3120`/`o3129` (Boletaria wall pieces), where even the lightmap ref is mislabeled under `obj/o3120` but every texture name still honestly starts `m03_`.

Together rules 3-4 resolve ~90% of the obj/parts references that don't already succeed via rule 1 or 2 (measured against every mounted obj/parts material). A FLVER0 texture entry's `ParamName` (the slot, e.g. `g_Diffuse`) is a separate optional field that's sometimes simply absent even though the path and the texture itself are both real — `InferMissingParamNames` (runs first, before any of the above) recovers these by positionally matching each untyped entry against a `Dif`/`Spc`/`Bmp`/`Lit`/`Dcl` token found in the MTD's own bracket tag, in the same order the entries already appear — validated 100% (83/83 real affected materials, 44 files spanning chr/map/obj/parts, including the `c9983`/`c9981` Ghost/Wanderer phantom-gear template) before being wired in.

**FLVER↔Godot coordinate/winding conventions** (already handled correctly in the importer — don't "fix" these again):
- **X is negated** on both position and normal: FLVER's coordinate convention is the mirror image of Godot's (confirmed via bind-pose FK on a character skeleton plus a map that visibly rendered as a mirror image of its sibling map — see `context.md`). Without this, every map/model/texture placement is a correct-looking but fully mirrored left/right image of the real thing — self-consistent enough (correct winding, correct lighting) to go unnoticed for a long time.
- **Triangle winding is swapped** (two indices per face) — *because of* the X negation above, not despite it. Godot uses **clockwise** front-face winding, the opposite of the CCW convention Blender/Wavefront OBJ expect; mirroring one axis flips the apparent winding, so the swap is what keeps faces front-facing after the negation, not a separate correction for the CW/CCW convention itself.
- No V-flip on UVs: `Image.CreateFromData` uses the same top-down row order as the raw decoded texture, so FLVER's raw V is already correct as-is.
- Mesh nodes must use `ImporterMesh`/`ImporterMeshInstance3D`, not the runtime `ArrayMesh`/`MeshInstance3D` — the scene-import pipeline's post-processing (LOD/shadow mesh generation, compression) only recognizes the import-time types; returning runtime types gets silently dropped on save.
- Any child added under the importer's returned root **must** have `.Owner` set to the root — `PackedScene.pack()` only serializes descendants with an owner; a plain `AddChild()` alone is invisible to the save step.
- Alpha handling: FLVER0's parsed material data exposes no blend-mode flag, only the MTD path string. `_Edge` in the MTD name → `AlphaScissor` transparency (foliage-style hard cutout); `_Alp` → `Alpha` blend (soft transparency, e.g. hair); `_Add` → `Alpha` transparency + `BlendMode = Add` (additive glow — volumetric light shafts, clouds, water shimmer; confirmed as a real convention across ~15 mounted map files, never used on solid geometry). This is the same naming convention used across FromSoft's other Souls titles.
- `g_Specular` (a Blinn-Phong spec map) is mapped onto Godot's PBR `MetallicTexture` slot as a deliberate simplification (semantically wrong, but the only slot available without a custom shader) — `Metallic` must be explicitly set to `1.0` since it multiplies against the texture and defaults to `0`.
- Terrain ground-blend materials (grass/dirt/stone transitions): a material whose MTD filename contains FromSoft's own `[M]`/`[ML]` blend-shader bracket tag (**not** whether `g_Specular_2`/`g_Bumpmap_2` are present — each is independently optional per layer; gating on their presence was tried first and was wrong, see `context.md`) gets a `ShaderMaterial` using `addons/archstone/terrain_blend.gdshader` instead of `StandardMaterial3D`. The two layers blend by `VertexColor.A` — **not** `.R`/`.G`/`.B` (R=G=B is a separate, still-unidentified per-vertex signal; confirmed *not* a shading/AO gradient either, despite briefly looking like one). Both wrong turns and their evidence are in `context.md` — don't re-try either. The second layer's own UV set is FLVER's UV channel 1 (Godot's native UV2 slot, via the `Mesh.ArrayType.TexUV2` array — see the mesh-building note below); UV channel 2 is reserved for the still-unread lightmap (see "Known deferred work"). Materials with `g_Diffuse_2` but no bracket tag are an unrelated "ghost" character dissolve effect, left on the single-layer path. Shader-only changes to `terrain_blend.gdshader` need no reimport — Godot loads `.gdshader` source live by path.

- Water surfaces (moats, swamps, lakes, fountains): gated on the material having a `g_Envmap` texture — confirmed unique to this family across all 8889 materials in the game, so unlike the other paths this needs no filename/MTD-tag heuristic at all. Gets a `ShaderMaterial` using `addons/archstone/water.gdshader`. These materials carry only `g_Bumpmap`+`g_Envmap`, no diffuse — `g_Envmap` is a small flat "impression" texture (confirmed by decoding and viewing one directly), used as a matcap-style reflection lookup, not a real cubemap. Per-material wave/reflection tuning (tile scale, scroll speed, water tint, Fresnel curve, opacity) has no equivalent in FLVER0's own material data, so this is the one path that reads the real `.mtd` file directly via `SoulsFormats.MTD.Read()` (indexed once at importer construction — see `_mtdIndex` in `FlverSceneImporter.cs`) instead of just the MTD path string; falls back to the shader's own built-in defaults if that file can't be resolved. `g_WaterColor` is a Float4 — its 4th component (`water_alpha`) blends the see-through screen-refraction toward a flat opaque tint, which is what actually separates murky/opaque water (e.g. swamps) from clear reflective water (e.g. moats); reading only its RGB and ignoring alpha was an earlier bug, see `context.md`. `water_alpha` is further scaled by a depth-based shore/deep fade (`g_WaterFadeBegin`, via `hint_depth_texture` + `INV_PROJECTION_MATRIX` reconstruction) so shallow edges stay see-through even on otherwise-murky water. `ROUGHNESS`/`SPECULAR` are derived from `g_ReflectBand` (confirmed real, varies `0`/`0.1` game-wide — `g_SpecularPower` was checked and is a constant `100` everywhere, not usable), adding a small *extra* real specular sun-glint on top. **The whole reflection/refraction/Fresnel result is written to `EMISSION`, not `ALBEDO`** (`ALBEDO` is left at `vec3(0)`) — this is a self-contained, hand-computed color the same way the original PS3 shader worked internally, not a physically-based surface meant to plug into Godot's lighting model; writing it to `ALBEDO` made it nearly invisible (Godot's PBR shifts energy from diffuse to specular as roughness drops, and there's no real reflection probe in-scene for a low-roughness "mirror" to reflect). The refraction sample's screen-space offset is `N.xy * refract_band`, scaled down and hard-clamped to a small constant (`refraction_scale = 0.03`, ±0.03 max) — `g_RefractBand`'s real range (`0.05`-`0.8`) is far too large to use directly as a raw `SCREEN_UV` delta (0-1 range); **this specific scale constant is an uncalibrated guess**, unlike everything else here, since there's no visibility into what unit the original shader actually used. The bump-sample coordinate (`g_TileScale_N`/`g_TexScroll_N`, tiling+scroll together) also has a flat `wave_detail_scale = 12.0` multiplier — applied to raw UV, the wave pattern tiled at ~40-95 *world units* per repeat on large meshes, user-confirmed as the actual cause of a "stretched, reflective-looking" wave pattern; **this constant is also an uncalibrated guess**. `a06_lava[we].mtd` also matches this family (`g_Envmap`-gated) and currently renders through this same shader, which is almost certainly wrong for lava — not yet addressed. **Visual accuracy tuning is deliberately paused, not abandoned or considered broken.** Coloring/depth-fade/tiling are working and were confirmed reasonably close against real screenshots (Defilement's brightness matched a sampled real reference to within a few percent); the remaining gap (Boletaria's water reads noticeably darker than reference footage even accounting for every real per-material MTD difference) traces to there being **no `WorldEnvironment`/tonemap/exposure configuration anywhere in this project** — checked directly, none exists — so comparisons so far have been against the 3D editor's own opaque default preview lighting, not a controlled baseline. Further water-shader tuning is on hold until real lighting/environment/ambience work happens (see `PLAN.md`) — see `context.md` for the full investigation.

**`SoulsFormatsNEXT/`** is a git **submodule** pointing at the user's own fork (`github.com/Solaire9886/SoulsFormatsNEXT`) of the community SoulsFormats library, not the upstream `soulsmods` repo directly — check `SoulsFormatsNEXT/FORMATS.md` for format documentation. Two local patches live as real commits on that fork (`origin` remote; `upstream` remote still points at `soulsmods/SoulsFormatsNEXT` for pulling future updates): one known upstream bug patched — `Formats/FLVER/FLVER0/Mesh.cs` had a `version >= 0x12`-based heuristic for picking a UV fixed-point scale factor (1024 vs 2048) that is wrong for Demon's Souls — verified empirically (UV wireframes overlaid on decoded textures) that Demon's Souls always needs `1024` regardless of header version; if SoulsFormatsNEXT is ever synced from upstream, re-check this hasn't regressed. The other patch retargets its own `.csproj` to `net8.0` to stay a valid `ProjectReference` target for `Soulbrandt.csproj` (also net8.0). Cloning this project fresh needs `git clone --recurse-submodules` (or `git submodule update --init` after a plain clone) to pull the submodule in.

**BND/DCX container unpacking needs no external tool.** `addons/archstone/AssetExtractor.cs` reads FromSoftware's containers directly in-process via `SoulsFormatsNEXT`'s `DCX`/`BND3`/`BND4` readers — see "Asset mounting" below. No third-party unpacker (WitchyBND, Yabber, etc.) is needed anywhere in this workflow; don't reach for one.

## Asset mounting

`mounted` (repo root, gitignored) is a real directory — **not a symlink anymore**. The toolbar's "Archstone" menu (`addons/archstone/archstone.gd`, `import.tscn`) splits this into three actions: "Mount..." only validates a chosen root (checks it contains at least one known category folder) and remembers it — it does **not** extract anything by itself; "Import" then offers a **Full Import** or **Choose Categories...** (checkboxes) confirmation, either of which kicks off the real extraction; "Clear Mounted Assets..." recursively deletes `res://mounted` after a confirmation dialog, for freeing disk space or forcing a clean re-extract (it does not also invalidate `.godot/imported/*` — that's the separate manual step in "Force a reimport" above).

The user points "Mount..." at their own raw, unmodified copy of the game (`.../Demon's Souls (USA)/PS3_GAME/USRDIR`); once "Import" runs, `addons/archstone/AssetExtractor.cs` walks it and unpacks `.bnd`/`.dcx` containers directly via `SoulsFormatsNEXT`'s `DCX`/`BND3`/`BND4` readers (confirmed against real containers: this game's compression is `DCX_EDGE`, and `SoulsFormatsNEXT` has a literal `Type.DemonsSouls = Type.DCX_EDGE` alias) — no external tool involved at all. Output goes straight into `res://mounted` in the same loose-file layout the importer already expects, so `FlverSceneImporter.cs` itself needed **no changes** for this. Re-running is idempotent (mtime-based skip, near-instant once already extracted) and scoped to a small category allowlist matching what the importer actually reads (`chr`, `map`, `obj`, `parts`, `mtd` — see `AssetExtractor.KnownCategories`; the raw dump's `movie`/`sound`/`msg`/`param`/etc are never touched). The chosen root path is remembered in `user://archstone_mount.cfg` (a `ConfigFile`, structurally outside the repo — never `res://override.cfg` or `EditorSettings`, see `context.md` for why).

Extraction is also runnable headlessly, independent of any editor GUI session: `godot-mono --headless --path . -s addons/archstone/extract_cli.gd -- --raw-root=/path/to/raw [--categories=chr,map]`. `AssetExtractor.cs` itself has zero `EditorPlugin`/`EditorInterface` dependency — both entry points call the exact same code. This is also the only way to exercise this feature in a GUI-less environment.

This is the editor-side increment only (`PLAN.md`'s Phase 1 item 1) — a full runtime loader for exported/shipped game builds (reading assets at play time with no editor and no pre-extraction step at all) is still future work, not started.

## Known deferred work

- Skeleton/animation import (HKX/Havok parsing, `Skeleton3D`/`AnimationPlayer` construction) — the stated long-term goal, not started.
- The FLVER vertex-format crash (see the known-errors note in "Build & verify" above) — a different vertex/UV buffer layout than the one currently handled. Originally only seen on a handful of map files; now also confirmed on at least one `obj` file (`o9996.flver`) now that `obj` is importable at all. Likely affects a similar fraction of `parts`, unconfirmed.
- `MetallicTextureChannel` only reads the texture's Red channel by default; a full-RGB spec map's G/B channels are silently dropped. Accepted ceiling, not currently worth chasing.
- Z-fighting/clipping when multiple whole maps are loaded together is presumed to be overlapping LOD variants of the same geometry (no LOD selection logic exists yet), not an importer bug — since strengthened from a guess to real evidence: `context.md`'s "map-piece naming investigation" confirms some map pieces (the `l1` suffix, isolated to Boletarian Palace so far) are genuine but *unplaced* LOD/alternate variants sitting right on top of their base counterpart's geometry, and more generally that bulk-importing every loose FLVER in a folder — which is exactly what happens today — doesn't distinguish "actually placed" from "unused leftover," which is the root of this.
- **Some `obj/` FLVER0 files genuinely contain zero meshes — this is correct, not a resolution failure.** Confirmed by scanning every mounted `obj/` FLVER0 directly: of 1068 files, 62 have no mesh. 57 of those are real, by-design attach/anchor markers (0 meshes, 0 materials, just 1-2 `Dummy` points and a single root `Node`, e.g. `o6511`/`o6600`) — the same attach-point convention (recurring refids like 191/192/200) also shows up on real meshed props (`o6700`'s `dmy_191`/`dmy_192`), so it's a generic game-wide convention, not unique to the empty files. The other 5 (`o2305`, `o2309`, `o2310`, `o4700`, `o6351`) are 288-byte bare-minimum stubs with nothing at all — read as allocated-but-never-built object IDs. Neither case is fixable by a smarter `CandidateDirs`-style rule, since there's no mesh anywhere to find.
- **Destructible-prop debris (a subset of `obj/`, e.g. `o6511`-`o6602`) is a genuinely unhandled system, not a missing resolution rule.** These pair a dummy-only FLVER0 (see above) with an `hkx` file and no `tex/` folder at all — verified directly against the raw, unmounted PS3 `.objbnd`/`.objbnd.dcx` (byte-identical inner content in both) that this is real shipped data, not an extraction artifact. The raw game root also has `map/breakobj/*.breakobj` (20 files, one per map area, magic header `OBJB`) — completely undocumented in `SoulsFormatsNEXT/FORMATS.md` and unparsed anywhere in this project. Working theory, not yet confirmed: these IDs are physics-only fracture anchors, and the actual visible debris is composed at runtime by the Havok destruction system from a small shared library of generic rubble chunks, driven by `.breakobj` data (placement + physics + which chunk to use) rather than stored per-object-ID. Would need `.breakobj` reverse-engineered (and likely `param`/`paramdef` data, currently outside `AssetExtractor`'s category allowlist entirely) before this could become an importer feature at all.
- `g_Lightmap` (a baked lighting texture) is present on 7160/8889 mounted materials (80%, 1223/1639 map files — 75%) and is currently never read at all. Deliberately scoped out of the terrain-blend fix above since it's much larger: `StandardMaterial3D` has no slot for an independent-UV multiply texture, so *every* material with a lightmap — not just the 217 terrain-blend files — will need the `ShaderMaterial` path once this is tackled. Its UV channel is always the *last* one in FLVER's per-vertex UV list (UV1 on non-blend/2-channel meshes, UV2 on blend/3-channel meshes) and is always confined to roughly [0,1] (a unique atlas unwrap), unlike the wide tiling ranges of diffuse UVs — confirmed via numeric UV-range analysis, not guessed. On non-blend materials Godot's native UV2 slot is free for it directly; on terrain-blend materials both native UV slots (UV, UV2) are already used by the two diffuse layers, so lightmap UV there will need a third per-vertex channel via the `Mesh.ArrayType.Custom0` array (pack u,v into a `Color`'s `.r,.g`), read back in-shader via `CUSTOM0.rg`.
